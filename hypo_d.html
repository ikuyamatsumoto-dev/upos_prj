HYPOTHESIS D

手入力が汚染するデータ品質
——入力ゼロ化がAI活用の前提条件

ユーポス × AI基幹連携｜データ品質・入力精度問題の仮説

Section 01 — Bottleneck Hypothesis

なぜAIが「使えない」のか——競合仮説と主軸仮説

基幹システムにデータが入っていても、そのデータが不完全・不統一・入力漏れの状態であれば、AIにとって意味をなさない。入力の問題はシステム問題より「見えにくい」ため、放置されやすい。

A

システム統合の問題

車両DB・顧客DBが分断されており、データ連携ができていない。（→ 仮説A）

B

次のアクション設計の問題

データはあっても「誰に何をすべきか」の判断ロジックがない。（→ 仮説B）

C

現場スキル・習慣の問題

ツールがあっても現場が使いこなせず、形骸化する。（→ 仮説C）

▶ 採用仮説
D

データ品質・入力精度の問題

入力が省略・誤記・未入力のままで、AIの学習・推論に使えるデータが実は揃っていない。

問題の根本は「入力コストの高さ」にある。商談後に手で履歴を打ち込む文化が続く限り、入力漏れは構造的に発生し、AIが参照するデータは永遠に"ゴミ入力・ゴミ出力"の状態を抜け出せない。

入力コストが高い——商談後に帰社してから5〜10分かけて手打ち入力
入力省略・抜け漏れ——忙しいときは「あとで」が「永遠にしない」になる
データ品質の劣化——顧客の買取希望額・来店理由・検討状況が欠落したまま蓄積
AIが機能しない——不完全データでは再来店予測・次回アプローチのレコメンドが不正確になる
「AIを入れても意味なかった」という評価——ツール問題ではなくデータ問題なのに、AIのせいにされる
Section 02 — Two-Step Approach

解決の2段階アプローチ

「入力をなくす」ことを先に達成しないと、AIへの投資は無駄になる。まず入力自動化でデータ品質を上げ、その後AIによるインサイト活用へ移る。

STEP A — 根本解消

音声・OCR・自動連携による
入力ゼロ化

商談録音をWhisperで自動文字起こし→AI要約→CRM自動登録。車検証はOCRで読み込み。「打つ」文化をなくす。

STEP B — スケール

蓄積されたクリーンデータで
AIインサイトを稼働

品質の揃ったデータが3〜6ヶ月蓄積された段階で、再来店予測・アプローチタイミングレコメンドを本格稼働させる。

Section 04 — Implementation Patterns

一般エンジニアが構築できる実装パターン

Pattern 01

Whisper API + Claude Haiku｜商談録音→AI要約→基幹入力自動化

スマホで商談を録音 → OpenAI Whisperで文字起こし → Claude Haikuで要約・構造化 → Googleフォーム or スプレッドシートに自動保存。プログラミング経験があれば数時間で構築可能。

構築時間数時間〜半日
月額コスト数千円〜
必要スキルPython基礎
ユーポス転用：査定商談（10〜30分）をスマホ録音→スタッフが打ち込む手間ゼロ。来店理由・希望金額・次回連絡希望日が自動でDBへ。1,000問/月あたり約500〜2,000円（Haiku利用時）。
参考：sophiate.co.jp — 中小企業向け音声→CRM自動化事例
Pattern 02

symphony 車検証OCR｜証書スキャン→即時DB登録

買取時に受け取る車検証・自賠責保険証を専用アプリでスキャンするだけで、車両情報が基幹DBに自動登録。手入力ゼロ・転記ミスゼロ。中古車業界向けに特化したSaaSで、初期設定は数日。

構築時間数日（設定のみ）
月額コスト要問合せ
必要スキルなし（ノーコード）
ユーポス転用：89店舗すべてで車検証入力が自動化される。買取1件あたり3〜5分の入力工数が消え、査定スタッフが顧客対応に専念できる。IT不慣れな店舗でも運用可能。
参考：symphony株式会社（中古車業界向けSaaS）
Pattern 03

Google Apps Script + Gemini API｜フォーム入力→AI補完→スプレッドシートDB

Googleフォームで必要最小限の入力（名前・車種・希望額のみ）→ Apps ScriptがGemini APIを呼び出して欠損項目を推定・補完→スプレッドシートに整形保存。入力フィールドを減らすことで入力漏れを構造的になくす。

構築時間1〜2時間
月額コスト無料〜数百円
必要スキルGAS基礎
ユーポス転用：「名前・車種・走行距離・希望額」の4項目だけ入力 → AIが「推定年式・市場相場・フォロー優先度」を自動付与。入力項目を減らすことで入力率が上がる逆転の発想。
参考：Google Workspace AI Assistant サンプル

Section 06 — Execution Roadmap

実行ロードマップ

1

現状調査：どのデータが欠落しているか把握する（1〜2週間）

基幹DB上の空白フィールド率・入力精度を本部が集計。「来店理由」「検討ステータス」「希望売却額」の欠落率を可視化する。

⚠ 障壁：データ抽出・分析できるエンジニアが必要。外部委託も選択肢。
2

1〜3店舗でパイロット：音声録音→AI要約を試す（2〜4週間）

Whisper+Claude Haikuの組み合わせを1店舗で試験導入。商談後の入力工数変化を測定する。

⚠ 障壁：スタッフへの録音習慣の定着が最大の課題。「録音=監視」と感じさせない説明が必要。
3

車検証OCR導入：入力の最低保証ラインを確立（1〜2ヶ月）

symphonyまたは同等のOCRサービスで車検証を自動取得。「車両データは必ず揃う」状態を作り、残りは音声入力で補う。

⚠ 障壁：既存基幹システムとのAPI連携が必要なケースあり。IT部門または開発ベンダーが対応。
4

データ品質レビュー：3〜6ヶ月のクリーンデータを確認（継続）

入力自動化導入後3〜6ヶ月で、欠落率・精度を再測定。AIインサイト活用フェーズへ移る準備が整っているか判断する。

⚠ 障壁：「品質が上がった」の定量基準を事前に定めておかないと判断できない。KPIを設計段階から決める。
5

AIインサイト本格稼働：再来店予測・アプローチタイミングを自動化（3〜6ヶ月後）

クリーンデータが揃った段階で、仮説B（次のアクション設計）と組み合わせたAIレコメンドを全店舗展開する。
